# Vendored Source

The `types/` directory is a vendored copy of the modular TypeScript definitions
generated by the B2C Commerce Script API documentation build.

| Field | Value |
|---|---|
| Upstream path | `digital/server/source/documentation/build/generated/scriptapi/typescript-modular` |
| Upstream version | `26.7.0` (see `types/upstream-package.json`) |
| Imported on | 2026-05-21 |

## Re-vendoring

When a new Script API release lands, regenerate the bundle from the digital repo
and replace `types/` wholesale:

```bash
rm -rf packages/b2c-script-types/types
cp -R \
  /path/to/digital/server/source/documentation/build/generated/scriptapi/typescript-modular \
  packages/b2c-script-types/types
# Drop validation artifacts that don't ship to consumers:
rm -rf packages/b2c-script-types/types/ts-validation-samples
rm -f packages/b2c-script-types/types/tsconfig.validate.json
rm -f packages/b2c-script-types/types/unresolved-generics.txt
# Keep upstream metadata under a different name so it doesn't clash with our package.json:
mv packages/b2c-script-types/types/package.json packages/b2c-script-types/types/upstream-package.json
mv packages/b2c-script-types/types/tsconfig.json packages/b2c-script-types/types/upstream-tsconfig.json
```

Update `version` in `package.json` and the table above.

## Why vendored

Until the digital docs-download servlet exposes the modular bundle alongside the
SFRA / Markdown docs, we ship a snapshot here so the VS Code extension and the
`b2c setup ide vscode-types` CLI command have something to point at. When the
servlet adds the bundle, this package can switch to fetching at install time
(or be replaced with a dynamic flow) without changing its consumer API.
